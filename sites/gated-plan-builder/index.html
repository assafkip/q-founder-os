THE PROBLEM

Review findings go to die in documents

Lost in docs

Expert reviews produce findings. Findings land in a Google Doc. The Google Doc gets skimmed once and archived. The product ships unchanged.

Fixed in demo, not in product

Someone manually patches the demo. The underlying code never changes. Next customer sees the same broken output. The fix was a bandage, not a gate.

No root cause

Reports say "we missed it." Nobody asks why the system missed it. Without root cause analysis, the same class of bug keeps appearing in new features.

HOW IT WORKS

7 steps from findings to gates

A structured pipeline that converts unstructured review feedback into a testable, dependency-aware engineering plan.

01

Catalog Findings

Every finding from every reviewer goes into a structured table. Source, impact, demo-fix status. Nothing slips through.

02

Group into Gates

Related findings cluster into coherent gates. Each gate has a clear scope, testable criteria, and addresses 1-5 findings.

03

Map Dependencies

Which gates can run in parallel? Which are sequential? The dependency graph tells you the fastest path through the plan.

04

Write Gate Specs

Each gate gets scope, implementation notes, root cause analysis, and 3-8 pytest test functions. Enough detail to execute without ambiguity.

05

Build Coverage Matrix

Every finding maps to a gate and a test function. Zero blank cells. If a finding has no resolution, the plan is not done.

06

Predict Persona Verdicts

Show what each reviewer's assessment changes to after all gates pass. "Needs edits" becomes "Deploy as-is." Concrete before/after.

07

Define Success Criteria

Specific, testable conditions for "done." Not "improve quality" but "all 6 persona verdicts change to Deploy or Approve."

FEATURES

What makes this different

TRACEABILITY

Finding-to-Gate Matrix

Every finding maps to either a manual fix or a permanent gate. The coverage matrix has zero blank cells. Nothing gets silently dropped.

TESTING

Pytest Gate Tests

Each gate ships with 3-8 pytest test functions. Tests describe the behavior they validate. Run the suite to prove the gate passed.

PARALLELISM

Dependency Mapping

An ASCII dependency graph shows which gates can run in parallel and which must be sequential. Ship faster by working on independent gates simultaneously.

DIAGNOSTICS

Root Cause Analysis

Every gate explains why the system missed the issue, not just that it was missed. Prevents the same class of bug from recurring in future work.

PREDICTIONS

Persona Verdicts

Before/after predictions for each reviewer persona. See exactly how each gate changes the assessment from "Needs edits" to "Deploy as-is."

CONTINUITY

Existing Gate Numbering

Continues from your project's current gate numbering instead of restarting from zero. Integrates with existing plans and test suites seamlessly.

Gate Specification

gate-8-context-aware-policies.md 
## Gate 8: Context-Aware Policy Generation

Addresses findings: #1, #4
Why these were missed: The policy generator treats all identity types
  identically. It has no concept of "account type" so it applies
  human-user defaults (AND logic, 90-day rotation) to service
  accounts and emergency access scenarios.
Depends on: None - can run in parallel with G9, G10

### Scope
  src/generators/identity_policy.py
  src/models/account_types.py (new)
  src/templates/conditional_access.jinja2

### Gate Test
File: tests/gates/test_gate8_context_aware.py

def test_emergency_ca_uses_or_logic():
    """CA policy for break-glass accounts uses OR, not AND."""

def test_service_account_rotation_365d():
    """Service accounts get 365-day rotation, not 90-day."""

def test_human_user_retains_90d_rotation():
    """Standard user accounts still get 90-day rotation."""

def test_account_type_enum_complete():
    """All 4 account types represented in enum."""

def test_policy_references_account_type():
    """Generated policy doc mentions account type in scope."""

QUALITY GATES

The plan must pass its own checklist

Before the plan is marked complete, it runs through these validation checks. The builder enforces all of them.

Every finding maps to either a demo fix or a product gate (no gaps in coverage matrix)
Every gate has 3-8 pytest test functions
Every gate specifies which files it touches
Dependencies are explicit (no circular dependencies)
Gates that can run in parallel are marked as parallel
Success criteria are specific enough to verify (not "improve output quality")
Gate numbering continues from existing project gates (never restarts from 0)
Root causes identified (not just "we missed it" but why the system missed it)
